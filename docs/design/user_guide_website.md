# Executor 使用手册网站规划

本文档定义 Executor 使用手册网站的目标、用户分层、信息架构、教程主线、接口教学顺序、示例质量约束和分阶段落地方案。它用于指导后续网站建设与内容编写，不替代现有的 `README`、`docs/API.md`、设计文档和版本记录。

具体任务拆分与验收清单见：[Executor 使用手册网站实施计划](../todolists/user_guide_website_plan.md)。

---

## 1. 背景

Executor 已经覆盖普通线程池任务、优先级、延迟与周期任务、批量提交、任务依赖、专用实时线程、GPU、跨线程通信、监控和失败可观察性。能力的完整度提高后，用户面临的新问题不再是“有没有接口”，而是：

- 第一次使用时，不知道最少需要理解哪些概念。
- 面对多个相近接口时，不知道当前场景应该选择哪一个。
- API 参考能够解释签名，却不能自然建立任务执行、生命周期、实时性和背压等心智模型。
- 高级能力与基础用法同时出现时，容易让只想快速提交任务的用户产生畏难感。
- 现有 README、API、设计、优化和计划文档分别服务不同目的，缺少一条由浅入深的连续学习路径。

网站的任务不是简单地把现有 `docs/` 搬到网页，而是把项目文档组织成一套渐进式学习产品。

---

## 2. 核心目标

网站应同时满足以下目标：

1. **几分钟内获得成功体验**：新用户复制少量代码即可运行第一个异步任务，不要求先理解线程池内部结构。
2. **通过业务问题引出接口**：先说明用户正在解决什么问题，再介绍对应 API，而不是按头文件声明顺序罗列功能。
3. **形成方案选择能力**：用户能够判断普通线程池、周期任务、实时线程、通信组件和底层逃生口各自适用的边界。
4. **支持按需深入**：入门用户可以随时停止阅读并开始工作；有实时性、性能或扩展需求的用户可以继续深入底层。
5. **保持示例可信**：教程代码尽量来自仓库内可编译、可测试的示例，避免文档与 API 演进脱节。
6. **中英文可持续维护**：第一阶段以中文内容建立结构，网站和文件布局为后续英文版本保留清晰映射。

网站的核心教学原则是：

> 从“提交第一个任务”开始，而不是从“理解线程池”开始；只有当业务出现新问题时，才引入下一项能力。

---

## 3. 非目标

第一阶段不追求：

- 在浏览器中编译和运行完整 C++ 项目。
- 替代 Doxygen 或手工维护一套重复的完整符号参考。
- 一次性重写所有历史设计、优化和发布文档。
- 在入门主线中讲完所有配置字段、状态字段和底层类。
- 对所有接口分配相同篇幅；常用接口进入主线，低频接口通过场景指南或参考页覆盖。

## 4. 用户分层

### 4.1 快速使用者

目标是尽快让任务在线程池运行。通常只需要：

- 构建和链接库。
- `Executor::instance()`。
- `submit()` 和 `future.get()`。
- 最基本的异常与退出语义。

这类用户不应在开始阶段接触调度器、无锁队列、CPU 亲和性或 Manager。

### 4.2 普通业务开发者

需要解决批量处理、优先任务、延迟重试、周期检查、任务依赖、状态监控等实际问题。重点是学习如何选择 Facade 接口，以及如何观察失败和正确关闭程序。

### 4.3 实时与并发系统开发者

关注周期确定性、线程间数据传递、背压、最新值语义、完整快照和严格阶段顺序。需要掌握实时执行器与 `executor::comm`，但仍优先从 Facade 使用项目。

### 4.4 框架集成者

需要独立 Executor 实例、资源隔离、自定义周期源、GPU 后端或对执行器生命周期做更细控制。此时开始进入公开的高级扩展接口。

### 4.5 性能与底层开发者

需要理解线程池、优先级调度、任务分发、负载均衡、工作窃取、无锁队列、对象池、实时线程循环以及相关 benchmark。

## 5. 内容分层

网站内容分为四层，每一层都允许用户在获得足够能力后停止继续深入。

### 第一层：十分钟上手

目标是建立最小成功路径：

1. 获取并构建项目。
2. 链接 `executor::executor`。
3. 使用 `submit()` 运行第一个任务。
4. 通过 `future.get()` 获取返回值和异常。
5. 说明懒初始化可以兜底，自定义配置必须在首次提交前完成。
6. 说明何时显式调用 `shutdown(true)`。

首个代码示例控制在可完整阅读的范围内：

```cpp
#include <executor/executor.hpp>
#include <iostream>

int main() {
    auto& ex = executor::Executor::instance();
    auto result = ex.submit([] { return 42; });

    std::cout << result.get() << '\n';
    ex.shutdown();
}
```

这一层不展示自定义线程数、实时线程、GPU 或监控配置。

### 第二层：连续场景教程

使用同一个小型机器人或数据处理流水线逐章演进。每一章只增加一个新的业务问题和一组必要接口，避免每章更换互不相关的玩具示例。

建议主线：

1. 后台解析一帧传感器数据：`submit()`。
2. 控制指令优先于普通分析：`submit_priority()`。
3. 设备暂不可用，稍后重试：`submit_delayed()`。
4. 周期采样与健康检查：`submit_periodic()`、状态查询和 `cancel_task()`。
5. 一批数据并行处理：`submit_batch()` 与 `submit_batch_no_future()`。
6. 规划必须等待感知完成：`submit_with_handle()`、`submit_after()`、`when_all()`。
7. 观察异常、提交拒绝与等待超时：failure callback、failure status、`wait_for_completion_ex()`。
8. CAN 控制需要专用周期线程：实时任务注册、启动、推送和停止。
9. 配置线程只需要把最新配置交给实时线程：`LatestMailbox<T>`。
10. 每条传感器消息都需要处理：`MpscChannel<T>` 或 `RealtimeChannel<T>`。
11. 监控线程读取完整状态：`DoubleBuffer<T>`。
12. 初始化、标定、运行具有严格顺序：`PhaseGate` 与 `Sequencer`。
13. 可并行计算转移到 GPU：GPU 注册、提交、降级和诊断。

每章保持统一结构：

1. 当前业务问题。
2. 默认推荐方案。
3. 最小可运行代码。
4. 预期输出或行为。
5. 为什么选择该接口。
6. 常见错误和失败如何观察。
7. 什么时候不该使用它。
8. 完整示例与下一章链接。

### 第三层：按问题选择方案

这部分不要求顺序阅读，用户可以从“我现在遇到什么问题”直接定位方案。

| 用户需求 | 默认选择 | 需要注意 |
|----------|----------|----------|
| 执行一次普通任务 | `submit()` | 用 `future.get()` 获取返回值和异常 |
| 少量任务有先后优先级 | `submit_priority()` | 优先级不等同于实时性保证 |
| 经过一段时间再执行 | `submit_delayed()` | 延迟调度属于 Facade，不是底层线程池接口 |
| 普通后台周期工作 | `submit_periodic()` | 适合软周期，不用于严格实时控制 |
| 固定周期且关注抖动 | 实时执行器 | 明确周期预算、背压和平台权限 |
| 一次提交很多任务 | 先 benchmark，再选择 batch API | 批量接口不承诺固定加速比 |
| 任务完成后触发后续任务 | `submit_after()` / `when_all()` | 依赖失败时后续任务默认不执行 |
| 只关心最新配置 | `LatestMailbox<T>` | 旧值被覆盖是设计语义 |
| 每条消息都需要消费 | `MpscChannel<T>` | 明确容量、关闭和满队列策略 |
| 实时周期内有限量消费消息 | `RealtimeChannel<T>` | 使用非阻塞 drain 和单周期预算 |
| 多读者读取完整状态 | `DoubleBuffer<T>` | 第一版适合单写多读 |
| 控制启动或阶段顺序 | `PhaseGate` / `Sequencer` | 超时和 missed phase 必须处理或观测 |
| CPU 与 GPU 自动选择 | `submit_auto()` | 需要说明后端可用性和回退语义 |
| 需要独立资源隔离 | 独立 `Executor` 实例 | 不与全局单例共享 Manager |

场景指南还应提供简短的决策树：

```text
任务是否要求严格周期？
├─ 否：使用普通 Facade 提交接口
│  ├─ 立即执行：submit / submit_priority
│  ├─ 稍后执行：submit_delayed
│  └─ 软周期：submit_periodic
└─ 是：使用实时执行器
   ├─ 只取最新状态：LatestMailbox
   └─ 消费有限消息：RealtimeChannel
```

### 第四层：高级逃生口与底层原理

Facade 是默认道路，高级接口只在用户遇到明确限制后引入。

高级逃生口按问题组织：

- **资源隔离**：使用独立 `Executor`，解释单例与实例模式的生命周期差异。
- **自定义周期源**：实现 `ICycleManager`，接管周期触发与外部时钟协同。
- **直接执行器控制**：使用 `ExecutorManager`、`IAsyncExecutor`、`IRealtimeExecutor` 和 `IGpuExecutor`。
- **高频单消费者任务聚合**：评估 `LockFreeTaskExecutor`。
- **GPU 资源控制**：进入 stream、内存池、统一内存、设备查询和调度器接口。
- **故障诊断**：深入状态计数、failure event、recent failures、通信事件和监控采样。
- **性能调优**：理解配置默认值、工作窃取、亲和性、批量提交和 benchmark 方法。

底层原理按一次任务的执行链路展开：

```text
Executor Facade
    -> ExecutorManager
        -> ThreadPoolExecutor
            -> PriorityScheduler
            -> TaskDispatcher / LoadBalancer
            -> WorkerLocalQueue / work stealing
            -> worker 执行与监控
```

实时路径单独说明：

```text
Executor Facade
    -> RealtimeThreadExecutor
        -> 周期回调
        -> 有界 MPSC 队列
        -> ObjectPool
        -> 单周期任务预算
        -> 背压与周期统计
```

## 7. 接口教学策略

### 7.1 Facade 优先

新手与普通业务章节默认只包含：

```cpp
#include <executor/executor.hpp>
```

底层类只在高级逃生口中出现。即使底层接口能够完成同一件事，主线也不提供两套并列方案，以免用户误以为必须理解它们才能使用项目。

### 7.2 先讲选择，再讲签名

每个接口首先回答：

- 它解决什么业务问题？
- 为什么默认选择它？
- 什么情况下换成另一个接口？

完整参数、字段和重载仍由 API 参考承担。

### 7.3 可靠性逐步出现

失败可观察性不能全部留到高级篇，但应渐进引入：

- 第一章：`future.get()` 会返回结果或重新抛出任务异常。
- 生命周期章节：初始化、懒初始化和 shutdown 语义。
- 周期与实时章节：返回值、状态计数、背压和超时。
- 可靠性专题：failure callback、recent failures、`ExecutorResult`、`WaitResult` 和监控。

### 7.4 明确软实时与严格实时边界

教程必须显式区分：

- `submit_periodic()`：普通后台周期调度，适合健康检查、刷新和采样等软周期任务。
- 实时执行器：独立周期线程，适合关注 jitter、CPU 亲和性、优先级和单周期预算的场景。

不得仅凭“定时执行”把所有用户都引向实时线程。

### 7.5 性能结论必须附带条件

批量提交、工作窃取、无锁路径和 GPU 加速都不能被描述为无条件更快。相关页面应链接可复现 benchmark，列出硬件、构建类型、任务规模和测量方法。

### 7.6 Facade 接口覆盖矩阵

“主线教程覆盖 Facade”不等于把所有接口堆进同一条学习路径。公开接口按学习价值分配到主线、专题、选型指南或参考页，确保完整覆盖但保持渐进披露。

| 接口组 | 主线教学位置 | 补充位置 |
|--------|--------------|----------|
| `Executor::instance()`、独立 `Executor` | 第一个任务、实例隔离 | 生命周期与高级逃生口 |
| `initialize()`、`initialize_ex()`、`shutdown()` | 初始化和关闭 | `_ex` 诊断与兼容 API 说明 |
| `submit()`、`submit_priority()`、`submit_delayed()` | 连续场景教程 | 提交接口选型指南 |
| `submit_periodic()`、`cancel_task()` | 周期采样章节 | 周期状态和失败专题 |
| `get_periodic_task_status()`、`get_all_periodic_task_status()` | 周期任务可观察性 | API 参考 |
| `submit_batch()`、`submit_batch_no_future()`、`submit_batch_priority()` | 批量处理章节 | benchmark 与性能选型 |
| `submit_with_handle()`、`submit_after()`、`submit_after_with_handle()`、`when_all()` | 任务依赖章节 | 失败传播与任务图边界 |
| `register_realtime_task()`、`start_realtime_task()`、`stop_realtime_task()` | 实时控制章节 | `_ex` 变体与配置参考 |
| `register_realtime_task_ex()`、`start_realtime_task_ex()` | 实时错误诊断 | `ExecutorResult` 参考 |
| `push_realtime_task()`、`try_push_realtime_task()` | 实时消息推送 | 背压和拒绝原因 |
| `get_realtime_task_list()`、`get_realtime_executor_status()` | 实时监控章节 | API 参考 |
| `get_realtime_executor()` | 不进入新手主线 | 底层实时执行器逃生口 |
| `get_async_executor_status()` | 状态与等待章节 | 状态字段参考 |
| `set_failure_callback()`、`get_failure_status()` | 可靠性主线 | failure event 参考 |
| `get_recent_failures()`、`clear_recent_failures()`、`set_recent_failure_capacity()` | 故障诊断专题 | 容量与长期运行建议 |
| `enable_monitoring()`、`set_monitoring_sampling_rate()` | 监控专题 | 开销与采样策略 |
| `get_task_statistics()`、`get_all_task_statistics()` | 监控专题 | 统计字段参考 |
| `wait_for_completion()`、`try_wait_for_completion()`、`wait_for_completion_for()` | 正确等待与关闭 | 兼容接口对比 |
| `wait_for_completion_ex()`、`is_idle()`、`get_completion_status()` | 可诊断等待 | `WaitResult` 与状态快照 |
| `register_gpu_executor()`、`submit_gpu()` | GPU 场景教程 | 后端与任务配置参考 |
| `register_gpu_executor_ex()` | GPU 后端诊断 | `ExecutorResult` 参考 |
| `get_gpu_executor_names()`、`get_gpu_executor_status()`、`get_all_gpu_executor_status()` | GPU 监控专题 | API 参考 |
| `get_gpu_executor()` | 不进入新手主线 | GPU 资源控制逃生口 |
| `submit_auto()` | CPU/GPU 选型指南 | 调度条件与回退语义 |
| `update_scheduler_config()`、`get_scheduler_config()` | GPU 高级调度 | 配置字段参考 |

`set_timer_thread_factory_for_test()` 是测试注入钩子，不属于用户使用手册，只在内部测试文档或 API 参考中标注其用途。旧 `bool` API 与可诊断 `_ex` API 不需要各写一套教程：主线优先选择能够表达错误原因的入口，兼容接口通过对照说明覆盖。

## 8. 示例代码策略

### 8.1 单一来源

教程的完整示例应存放在仓库中，例如：

```text
examples/tutorial/
  01_first_task.cpp
  02_priority.cpp
  03_delayed_periodic.cpp
  04_batch.cpp
  05_dependencies.cpp
  06_observability.cpp
  07_realtime.cpp
  08_communication.cpp
  09_gpu.cpp
```

网站中的较长代码尽量引用或同步生成自这些文件。无法自动引用时，也要通过对应测试或编译目标保证片段不失效。

### 8.2 示例验收要求

每个主线示例必须：

- 能够独立编译。
- 明确所需构建选项。
- 不依赖未说明的外部服务或硬件；GPU 示例提供无设备时的预期行为。
- 输出稳定且适合用户核对。
- 展示必要的错误处理，但不让样板代码淹没核心概念。
- 进入 CMake 构建；适合自动执行的示例进入 CTest smoke test。

### 8.3 页面呈现

每个示例页面至少提供：

- 核心片段。
- 完整源文件链接。
- 编译与运行命令。
- 预期输出。
- “代码发生了什么”的简短解释。
- 下一步推荐阅读。

第一阶段不要求浏览器内运行 C++。可以提供一键复制、完整文件下载和可选的 Compiler Explorer 链接。

现有文档继续作为事实来源，网站增加面向学习的组织层：

## 10. 技术方案建议

第一阶段建议使用 **VitePress**：

- Markdown 优先，适合复用现有文档。
- 静态构建，容易部署到 GitHub Pages。
- 支持导航、侧边栏、代码高亮、本地搜索和中英文目录。
- 可以在需要时使用 Vue 组件实现接口选择器或交互式决策树。
- 工具链相对轻量，不会让 C++ 库长期承担过重的网站维护成本。

建议的网站源码布局：

```text
website/
  package.json
  docs/
    .vitepress/
      config.mts
      theme/
    zh/
      index.md
      getting-started/
      tutorial/
      guides/
      advanced/
    en/
      index.md
```

最终布局应在脚手架阶段结合 GitHub Pages 路径、版本策略和现有 `docs/` 复用方式确认。第一阶段可以只发布中文，并确保路由结构允许后续增加 `/en/`。

## 11. 搜索、版本与多语言

### 11.1 搜索

搜索结果应优先区分：

- 教程：告诉用户怎么完成一个场景。
- 指南：帮助用户选择方案。
- 参考：提供完整签名和字段。
- 原理：解释内部工作方式。

页面标题和描述中应包含用户可能搜索的自然语言，例如“周期任务还是实时线程”“队列满了怎么办”，而不只写类型名。

### 11.2 版本

首版网站只维护当前开发版本，页面显著显示对应库版本。发布流程稳定后，再评估是否保留最近稳定版本文档。不要在尚未形成兼容策略前引入复杂的多版本插件。

### 11.3 多语言

中文先行，但从一开始保持中英文页面一一映射的目录习惯。代码示例、命令和 API 名称共用；解释文本分别维护。每个版本发布前检查 README、核心快速开始和迁移页是否同步。

## 12. 内容质量标准

每个页面发布前检查：

- 页面开头是否说明用户完成后能获得什么能力。
- 是否先给默认方案，再介绍可选方案。
- 是否只有当前章节必要的新概念。
- 示例是否来自当前公开 API，并经过编译验证。
- 是否说明失败、超时、背压或生命周期中的相关风险。
- 是否明确适用场景和不适用场景。
- 是否提供下一步阅读链接，但不强迫用户继续深入。
- 性能数据是否可复现并注明测试条件。
- 是否避免把内部实现细节写成稳定 API 契约。

## 13. 分阶段实施

### 阶段 A：内容骨架

1. 确定网站名称、域名路径和首页文案。
2. 建立完整导航与空页面骨架。
3. 完成“十分钟上手”三到五篇核心内容。
4. 在 README 中增加网站入口。

验收标准：第一次接触项目的用户能够从首页进入教程，完成构建、提交任务、读取结果和正确退出。

### 阶段 B：Facade 主线教程

1. 建立 `examples/tutorial/`。
2. 完成普通、优先级、延迟、周期、批量和依赖任务章节。
3. 完成失败观察和等待章节。
4. 将所有示例加入编译或 smoke test。

验收标准：主线覆盖常用 Facade 接口，用户无需接触 Manager 或底层执行器。

### 阶段 C：实时、通信与 GPU

1. 完成普通周期与实时线程选型指南。
2. 完成实时任务的注册、启动、推送、停止和背压观察。
3. 完成 Channel、Mailbox、RealtimeChannel、DoubleBuffer、PhaseGate 和 Sequencer 场景。
4. 完成 GPU 可选路径、后端探测和安全降级说明。

验收标准：实时和通信用户能够根据数据语义选择组件，并能观察队列满、覆盖、超时和 missed phase。

### 阶段 D：高级逃生口与原理

1. 完成独立实例、Manager、接口与 `ICycleManager`。
2. 按实际执行路径讲解线程池和实时线程内部结构。
3. 整理无锁队列、对象池、工作窃取和性能测试方法。
4. 建立从高级页面到设计、优化和 benchmark 原始文档的链接。

验收标准：高级用户知道何时离开 Facade，以及进入底层后需要承担哪些生命周期、并发和平台责任。

### 阶段 E：发布自动化与持续维护

1. 配置网站构建检查。
2. 配置 GitHub Pages 部署。
3. 增加链接检查和代码片段验证。
4. 将文档同步检查加入版本发布清单。

验收标准：代码/API 改动导致示例失效时能够在 CI 中被发现，网站可以随主分支或发布版本稳定部署。

## 14. 首批页面建议

首个可发布版本不需要等待全部内容完成，建议优先交付：

1. 首页。
2. Executor 是什么。
3. 安装与构建。
4. 第一个任务。
5. 返回值与异常。
6. 初始化和关闭。
7. 如何选择任务提交接口。
8. 普通周期还是实时线程。
9. 如何选择通信组件。
10. Facade 与高级逃生口。

这十页已经能够建立网站的核心价值：让新用户快速开始，让有具体问题的用户找到方向，让高级用户看到继续深入的入口。

## 15. 成功标准

网站完成第一阶段后，应能够用以下问题检验：

- 新用户能否在十分钟内运行第一个任务？
- 用户是否能在不阅读底层实现的情况下正确选择常用 Facade 接口？
- 用户能否判断 `submit_periodic()` 与实时执行器的区别？
- 用户能否根据“最新值、每条消息、完整快照、阶段顺序”选择通信组件？
- 用户是否知道 `future`、返回值、状态和 failure event 各自承担什么可观察性职责？
- 高级用户是否能明确找到 Manager、底层接口、自定义周期源和无锁结构入口？
- 教程示例是否随仓库一起编译并在 API 变化时及时失败？

如果这些问题可以稳定得到肯定答案，网站就不只是更好看的文档，而是真正降低了 Executor 的学习和使用成本。
